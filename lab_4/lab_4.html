1.- Espacio de direcciones de un proceso

Examinar el espacio de direcciones virtuales de un proceso en Unix es relativamente fcil. Esto es as porque el mapa de direcciones de cada proceso est representado como un archivo. Todos estos archivos se encuentran dentro del sistema de ficheros de la CPU, que en Linux est montado en la carpeta /proc. Este sistema de ficheros contiene toda la informacin pertinente a la CPU y sirve para configurarla.

Para poder gestionar y acceder a la informacin de los procesos de forma ordenada, /proc contiene una serie de directorios cuyo nombre (siempre numrico) es el PID de cada uno de los procesos que estn en ejecucin. Dentro de estos directorios puede encontrarse la informacin del proceso en cuestin. De especial inters para esta prctica es el archivo de nombre maps, que nos detalla el espacio de direcciones del proceso.

Por ejemplo, el archivo /proc/1234/maps contendr el espacio de direcciones del proceso 1234, y podramos ver su contenido mediante la orden cat:

$ cat /proc/1234/maps

Adems de los directorios cuyo nombre es un nmero, dentro de /proc tambin hay otro directorio especial, de nombre self que apunta siempre al proceso que se est ejecutando en un momento dado. Es, por tanto, un enlace simblico al proceso que accede a este archivo. Por tanto, si escribimos:

$ cat /proc/self/maps

Veremos en el terminal el espacio de direcciones del proceso cat, que el que se est ejecutando. Al hacerlo obtendremos un resultado estructurado en forma de tabla de seis columnas:

  • Direcciones: rango de direcciones virtuales
  • Protecciones y modo de comparticin: r si se puede leer, w si se puede escribir, x si se puede ejecutar, s si est compartido con otros procesos, y p si es privado.
  • Desplazamiento: si la regin de memoria ha sido mapeada desde un fichero, este es el desplazamiento (offset) en el fichero donde el mapeo empieza. Si la memoria no ha sido mapeada desde un fichero, el valor es 0.
  • Dispositivo: Si la regin ha sido mapeada desde un fichero, este campo indica los nmeros mayor y menor del dispositivo.
  • Inode: Si la regin ha sido mapeada desde un fichero, este es su inodo.
  • Nombre de fichero: Si la regin ha sido mapeada desde un fichero, este es el nombre del fichero. El campo se deja en blanco en caso de que la regin no haya sido mapeada desde un fichero (regin annima). Tambin existen regiones especialmente importantes que reciben nombres especiales, como [heap] o [stack].

Fjate en que este espacio de direcciones no tiene por qu incluir slo el programa originalmente invocado, sino tambin incluye el cargador dinmico, las bibliotecas dinmicas usadas, etc. Este mapa de direcciones presenta todo aquello que el proceso pueda ver.

2.- Seguridad en el espacio de direcciones

Los sistemas operativos modernos suelen incluir mecanismos de proteccin para evitar que un proceso pueda escribir en zonas de memoria sobre las que no se tienen permisos. Estos mecanismos de proteccin no deben crear falsas expectativas en el usuario: C no realiza comprobacin acerca de los lmites de los arrays, por lo que salirse de los lmites de un array no implica que necesariamente vayamos a obtener un error!

Para ilustrar los mecanismos de proteccin que proporciona el sistema operativo, considera el programa test_escritura.c. Este programa acepta como parmetro una direccin de memoria especificada en decimal, en octal (si empieza por 0), o en hexadecimal (si empieza por 0x). A continuacin, intenta realizar una escritura en la direccin de memoria indicada. Si no hay ningn error, se concluye que podemos escribir en dicha direccin de memoria.

test_escritura.c:


            #include <unistd.h>
            #include <stdio.h>
            #include <stdlib.h>
            
            char dato;
            char* dir;
            
            int main(int argc, char *argv[]) {
                if (argc != 2) {
                    fprintf(stderr, "%s: se necesita una direccion\n", argv[0]);
                    exit(1);
                }
                // obtenemos la direccin de memoria en donde intentaremos 
                // escribir de los argumentos de lnea de comandos.
                dir  = (char*)strtoul(argv[1], NULL, 0);
                printf ("Probando a escribir en la direccin virtual %p: ", dir);
                fflush(stdout);
                *dir = dato;
                // Si llegamos hasta aqu, no ha habido fallos y tenemos permisos de acceso
                printf("escribible\n");
                sleep(10);
                exit(0);
            }

2.1 Ataques de buffer y stack overflow

Para entender la importancia que tienen los mecanismos de proteccin de memoria es ilustrativo estudiar uno de los ataques informticos ms conocidos: el buffer overflow. Concretamente, estudiaremos los desbordamientos de pila (stack overflow).

El buffer overflow se aprovecha de que C no comprueba los lmites de los arrays para sobrepasar dichos lmites y escribir en otras zonas de memoria. Si la zona de memoria sobreescrita contiene cdigo ejecutable o almacena direcciones de salto es posible alterar el flujo de funcionamiento del programa. Cuando las variables implicadas en el ataque se encuentran en la pila hablamos de stack overflow. La siguiente figura intenta ilustrar este tipo de ataque:

Debemos aclarar que realmente es posible hacer que C realice comprobaciones sobre los lmites de los arrays usando las opciones de GCC. Evidentemente, habilitar estas comprobaciones tendr un impacto importante en el rendimiento del programa. Afortunadamente, los sistemas operativos modernos proporcionan otros mecanismos alternativos que previenen este tipo de ataques. Dos de los ms importantes son:

  • Aleatorizacin del layout de memoria (ASLR, Adress Space Layout Randomization). Para evitar que un atacante pueda saltar a una zona de memoria concreta tras realizar un ataque, ASLR aleatoriza las posiciones de memoria de algunas zonas clave de la memoria, como la pila (stack) y el montn (heap). Ejecuta y comprueba el /proc/XXX/maps de cualquier proceso un par de veces y comprueba este hecho.
  • Uso de canarios (canaries, en analoga con el uso de canarios en las minas): son valores conocidos que se colocan entre el buffer y antes de la zona de memoria que se quiera proteger. Gracias a estos valores, es posible monitorizar posibles desbordamientos de buffer.
