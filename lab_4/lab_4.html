1.- Espacio de direcciones de un proceso

Examinar el espacio de direcciones virtuales de un proceso en Unix es relativamente fácil. Esto es así porque el mapa de direcciones de cada proceso está representado como un archivo. Todos estos archivos se encuentran dentro del sistema de ficheros de la CPU, que en Linux está montado en la carpeta “/proc”. Este sistema de ficheros contiene toda la información pertinente a la CPU y sirve para configurarla.

Para poder gestionar y acceder a la información de los procesos de forma ordenada, “/proc” contiene una serie de directorios cuyo nombre (siempre numérico) es el PID de cada uno de los procesos que están en ejecución. Dentro de estos directorios puede encontrarse la información del proceso en cuestión. De especial interés para esta práctica es el archivo de nombre “maps”, que nos detalla el espacio de direcciones del proceso.

Por ejemplo, el archivo “/proc/1234/maps” contendrá el espacio de direcciones del proceso 1234, y podríamos ver su contenido mediante la orden cat:

$ cat /proc/1234/maps

Además de los directorios cuyo nombre es un número, dentro de “/proc” también hay otro directorio especial, de nombre “self” que apunta siempre al proceso que se esté ejecutando en un momento dado. Es, por tanto, un enlace simbólico al proceso que accede a este archivo. Por tanto, si escribimos:

$ cat /proc/self/maps

Veremos en el terminal el espacio de direcciones del proceso cat, que el que se está ejecutando. Al hacerlo obtendremos un resultado estructurado en forma de tabla de seis columnas:

  • Direcciones: rango de direcciones virtuales
  • Protecciones y modo de compartición: “r” si se puede leer, “w” si se puede escribir, “x” si se puede ejecutar, “s” si está compartido con otros procesos, y “p” si es privado.
  • Desplazamiento: si la región de memoria ha sido mapeada desde un fichero, este es el desplazamiento (offset) en el fichero donde el mapeo empieza. Si la memoria no ha sido mapeada desde un fichero, el valor es 0.
  • Dispositivo: Si la región ha sido mapeada desde un fichero, este campo indica los números mayor y menor del dispositivo.
  • Inode: Si la región ha sido mapeada desde un fichero, este es su inodo.
  • Nombre de fichero: Si la región ha sido mapeada desde un fichero, este es el nombre del fichero. El campo se deja en blanco en caso de que la región no haya sido mapeada desde un fichero (región anónima). También existen regiones especialmente importantes que reciben nombres especiales, como [heap] o [stack].

Fíjate en que este espacio de direcciones no tiene por qué incluir sólo el programa originalmente invocado, sino también incluye el cargador dinámico, las bibliotecas dinámicas usadas, etc. Este mapa de direcciones presenta todo aquello que el proceso pueda ver.

En este caso la primera región contiene el código del programa (segmento .text), con permisos de lectura y ejecución (r-xp), extraída del comienzo (dirección 00000000) del fichero “/bin/cat” (inode 812115 del dispositivo 08:05).             #include <stdio.h>
            #include <unistd.h>

                char* p1 = malloc(0x1000);
                char* p2 = malloc(0x1000);
                printf("\n");
                escribe("main", main);
                escribe("e1", &e1);
                escribe("e2", &e2);
                escribe("e3", &e3);
                escribe("e3[0]", &e3[0]);
                escribe("m", &m);
                escribe("m[0]", &m[0]);
                escribe("p1", &p1);
                escribe("p2", &p2);
                escribe("*p1", p1);
                escribe("*p2", p2);
                sleep(1000);
            }

2.- Seguridad en el espacio de direcciones

Los sistemas operativos modernos suelen incluir mecanismos de protección para evitar que un proceso pueda escribir en zonas de memoria sobre las que no se tienen permisos. Estos mecanismos de protección no deben crear falsas expectativas en el usuario: ¡C no realiza comprobación acerca de los límites de los arrays, por lo que salirse de los límites de un array no implica que necesariamente vayamos a obtener un error!

Para ilustrar los mecanismos de protección que proporciona el sistema operativo, considera el programa “test_escritura.c”. Este programa acepta como parámetro una dirección de memoria especificada en decimal, en octal (si empieza por “0”), o en hexadecimal (si empieza por “0x”). A continuación, intenta realizar una escritura en la dirección de memoria indicada. Si no hay ningún error, se concluye que podemos escribir en dicha dirección de memoria.

test_escritura.c:


            #include <unistd.h>
            #include <stdio.h>
            #include <stdlib.h>
            
            char dato;
            char* dir;
            
            int main(int argc, char *argv[]) {
                if (argc != 2) {
                    fprintf(stderr, "%s: se necesita una direccion\n", argv[0]);
                    exit(1);
                }
                // obtenemos la dirección de memoria en donde intentaremos 
                // escribir de los argumentos de línea de comandos.
                dir  = (char*)strtoul(argv[1], NULL, 0);
                printf ("Probando a escribir en la dirección virtual %p: ", dir);
                fflush(stdout);
                *dir = dato;
                // Si llegamos hasta aquí, no ha habido fallos y tenemos permisos de acceso
                printf("escribible\n");
                sleep(10);
                exit(0);
            }

2.1 Ataques de buffer y stack overflow

Para entender la importancia que tienen los mecanismos de protección de memoria es ilustrativo estudiar uno de los ataques informáticos más conocidos: el buffer overflow. Concretamente, estudiaremos los desbordamientos de pila (stack overflow).

El buffer overflow se aprovecha de que C no comprueba los límites de los arrays para sobrepasar dichos límites y escribir en otras zonas de memoria. Si la zona de memoria sobreescrita contiene código ejecutable o almacena direcciones de salto es posible alterar el flujo de funcionamiento del programa. Cuando las variables implicadas en el ataque se encuentran en la pila hablamos de stack overflow. La siguiente figura intenta ilustrar este tipo de ataque:

Debemos aclarar que realmente es posible hacer que C realice comprobaciones sobre los límites de los arrays usando las opciones de GCC. Evidentemente, habilitar estas comprobaciones tendrá un impacto importante en el rendimiento del programa. Afortunadamente, los sistemas operativos modernos proporcionan otros mecanismos alternativos que previenen este tipo de ataques. Dos de los más importantes son:

  • Aleatorización del layout de memoria (ASLR, Adress Space Layout Randomization). Para evitar que un atacante pueda saltar a una zona de memoria concreta tras realizar un ataque, ASLR aleatoriza las posiciones de memoria de algunas zonas clave de la memoria, como la pila (stack) y el montón (heap). Ejecuta y comprueba el “/proc/XXX/maps” de cualquier proceso un par de veces y comprueba este hecho.
  • Uso de canarios (canaries, en analogía con el uso de canarios en las minas): son valores conocidos que se colocan entre el buffer y antes de la zona de memoria que se quiera proteger. Gracias a estos valores, es posible monitorizar posibles desbordamientos de buffer.
